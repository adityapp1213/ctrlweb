Monarch ;

Monarch is Atom's thought-grounded model family. We believe future AI systems should reason through problems, understand context, and learn from interactions instead of relying only on pattern matching. Thought-grounded intelligence combines multimodal understanding and world knowledge to make AI more reliable, adaptable, and useful in real applications. This post walks through the architecture end to end, including how TGMI-Tiny forms thoughts, remembers, reasons, and handles math and code while staying small enough to run on a single consumer GPU.

The scale I'm working with

Everything here is built to run on a single RTX 4090 with 8GB of dedicated VRAM. That number shapes almost every decision below: how many specialists the workspace has, how big the memory buffer is, how long the context window can be. I'm running everything in mixed precision, checkpointing gradients through the heavier recurrent parts, and using an 8-bit optimizer to keep memory down, with a small per-step batch and gradient accumulation to make up the difference.

Thought formation

Before any reasoning happens, Monarch turns whatever it received into a single 512-dimensional vector, the thought. This isn't a one-shot projection. A small recurrent unit refines that vector against the input several times in a row, and a gate decides when the vector has settled. Easy, familiar inputs settle in a couple of steps. Unusual ones take longer, up to a cap of twelve refinement steps.

The hard part isn't forming a thought, it's stopping all thoughts from drifting toward the same point, which is the failure mode that quietly ruins a setup like this. The fix is a regularizer that runs underneath nearly every training signal in this architecture: it keeps the spread of thought vectors in a batch from shrinking, and keeps individual dimensions of the vector from becoming redundant with each other. It's a small piece of math, but it's the thing that keeps the whole idea from collapsing into a single useless point.

Four ways in, one shared space

Four kinds of input land in that same 512-dimensional space. Text goes in directly through the embedding table. Images and audio go through CLIP and Whisper's encoder, both are pretrained, both stay frozen, and I run them once ahead of time and cache the results so they never have to sit in GPU memory during training. Code and math are different. There's no off-the-shelf encoder for what a piece of code or an equation actually means structurally, so I built a small one from scratch, it parses the expression into a tree and reads it bottom-up, which keeps things like nesting and operator order intact instead of letting them blur together the way a flat sequence of tokens would.

A separate training signal pulls matching ideas from different inputs toward each other in this space, a caption and the photo it describes, an English description of a derivative and the parsed expression for it.

Memory, on different timescales

One vector, the self vector, carries forward across a conversation. It updates after every response, but only when something seems worth keeping, small talk doesn't overwrite it, a genuinely new piece of information does. That vector is also what the model uses to look things up: it's the query that goes out to memory whenever the model needs context beyond what's directly in front of it.

Memory itself isn't one flat pool. What just happened, what's been said in this conversation, what happened recently, and what the model has learned more durably are kept at different levels, because they shouldn't be searched or weighted the same way.

Five signals that shape how it reasons

Alongside the thought itself, the model also scores how it relates to that thought, on five continuous scales: how positive or negative it seems, how much it warrants engagement, how unfamiliar it is, how socially relevant it is, and how much it looks like something that needs care. These aren't labels attached to the output afterward, they change what happens next. The clearest example is compute: how aroused the model is about a problem directly scales how many reasoning passes it's allowed to spend on it.

Fig. 5 ; What each signal actually changes

Specialists that compete for attention

The model has eight specialist sub-networks, each leaning toward a different kind of reasoning, spatial, causal, language, social, abstract, planning, math, and code. For every thought, all eight score how relevant they are to it. The three most relevant win, and instead of working in isolation they attend to each other directly before handing off a combined result to the reasoning core. A visual puzzle pulls in a different trio than a social dilemma or a planning problem does. Math and code aren't built any differently from the other six, they're the same size, same structure. They become good at their domain the same way any of the others do, by being the ones that keep winning on that kind of problem during training.

Fig. 6 ; Specialists compete, then share

Reasoning: running the same stack more than once

Once the workspace has produced a combined context, the reasoning core runs against it, and runs again, and again, using the exact same weights each time, for as many passes as the problem seems to warrant. A self-check after the final pass decides whether the result is good enough. If it isn't, the model resets to the original input and tries once more, this time nudged by its own sense of what it should adjust. That single retry is the closest thing this design has to noticing a mistake and correcting it.

Getting math and code right

This is the part I spent the most time getting right, because the obvious approaches don't actually work well at this scale.

The first instinct is to treat math and code the way images and audio are treated, give them their own dedicated, pretrained encoder. That doesn't hold up, because images and audio need grounding in what something physically looks or sounds like, and there's no equivalent for code, what matters there is what an expression structurally is and what it does when evaluated, not a sensory pattern. That's why the code/math encoder described earlier is trained from scratch rather than borrowed: there's nothing pretrained that solves this particular problem.

A second instinct is to give the model an explicit, external map of mathematical and programming knowledge, a graph of theorems and techniques with their dependencies spelled out, retrieved and combined for every relevant problem. This is a real and well-studied approach, but it's heavy: it needs a large, carefully curated graph, and a fair amount of extra machinery to retrieve from it and combine the results at inference time. For a model this size, that's a lot of moving parts for the return.

What I actually use is simpler, and reuses machinery that's already in the architecture. A math or code thought is, most of the time, mostly a label for a technique, integration by parts, binary search, induction, and that label barely changes with the specific numbers in the problem. So I extract a flat list of a few hundred recurring techniques from worked solutions and tagged code problems, and apply the same collapse-prevention training signal from thought formation a second time, grouped by technique instead of by input type. Problems that use the same technique get pulled toward the same region of the thought space; the model ends up with something like a library of approaches, but it's geometry that emerges from training rather than a graph anyone had to build by hand.

Fig. 7 ; Techniques cluster on their own

Where exact answers actually come from

None of this is meant to make the model good at long division. A model this small isn't going to reliably multiply six-digit numbers in its head, and I don't think it should try. Instead, when the model needs an exact number or needs to check whether a piece of code actually runs, it writes out a short expression, hands it to a real calculator or a sandboxed Python process, and reads the answer back into its own context before continuing. The model's job is recognizing when to reach for this and what to compute, the arithmetic itself happens outside the network entirely, where it's exact by construction.

How I'm training it

Standard backpropagation covers almost the entire model. The one place it doesn't reach cleanly is the self-improvement step, judging whether a change the model proposed to itself actually helped over time isn't something a gradient can point to directly, so that one piece is trained with a simple reward signal instead, kept deliberately small and isolated from the rest of the network. Everything related to learning a technique quickly from only a few examples, the math and code procedures, and the visual puzzle patterns, uses a lightweight meta-learning approach: adapt briefly to a handful of examples, then nudge the starting point slightly toward wherever that adaptation landed.

What I expect, and what I don't

Nothing here has been trained yet, so everything below is a prediction, not a result. I expect tool use to matter a lot for grade-school and competition-style math, matter a fair amount for code, and matter not at all for visual abstraction puzzles, where the bottleneck isn't arithmetic at all.

BenchmarkWithout a calculatorWith oneWhy
Grade-school math25–35%55–70%reasoning picks the operation, the sandbox does the arithmetic
Short coding problems5–10%20–35%write, run, see the error, fix it, repeat a few times
Visual abstraction puzzles15–25%15–25%no amount of arithmetic help fixes a pattern it hasn't recognized

Where I don't expect this to be strong: recalling obscure facts (there's just not enough capacity to memorize broadly at this size), holding long documents word-for-word, generating highly formulaic text where pattern-matching already wins, and raw speed, refining a thought, running a specialist competition, and reasoning over multiple passes all add latency a plain model the same size wouldn't pay.

Building it

I'm building this in dependency order, the input encoders and the thought refiner first, tested in isolation before any real training data touches them, then memory and the emotional signals, then the workspace, then reasoning, then the full model wired together and checked for memory fit on the actual card before any real training run starts. The calculator sandbox gets its own isolated tests before anything in the model is allowed to depend on it. Every stage gets a short trial run first, checked against a monitoring script, before I scale it up, a collapsing thought space looks identical to normal training for a while before it's obviously broken, so I'm watching for it from the very first step, every stage, no exceptions.

I'm planning to lean on Claude Code and Codex heavily for the implementation itself, scaffolding the modules, writing the training loops, chasing down shape mismatches and memory errors. The judgment calls stay with me: whether something is collapsing, whether a stage actually converged, whether the technique clustering shows real signal. That part doesn't get automated.

Closing thought

Most of what holds this together is one mechanism reused in several places rather than a pile of separate ideas. The same collapse-prevention signal trains thought formation, multimodal grounding, and technique clustering. The same halting pattern shows up in thought formation and in reasoning. The same meta-learning loop handles visual puzzles and math techniques without needing to know which one it's looking at. None of it is exotic, the goal was never novelty for its own sake, just an architecture small enough to actually finish training on the hardware I have.
